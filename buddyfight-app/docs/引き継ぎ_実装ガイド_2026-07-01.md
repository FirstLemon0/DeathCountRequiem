## 1. カード実装の手順（最重要）

新製品（例: 次のブースター）を実装する標準フロー。既存製品はこの手順で 8製品→ EB01 まで実装済み。

### 1-1. データ取得（公式から）
- `tools/fetch_card_data.py` を使う。`--expansions` で expansion 番号↔製品名、`--expansion <N> --out <file>` で全カード（スタッツ・属性・効果テキスト全文）を取得。詳細は [[bf-card-data-fetch-method]] とツール冒頭 docstring。
- expansion 番号の例: 24=SD01, 25=SD02, 26=BT01(ギガ・フューチャー), 27=EB01(ウルトラ必殺パック), 28=TD01, 29=TD02, 30=BT02(ギャラクシー・バースト), 31=EB02(Wヒーロー大戦), 32=EB03(百雷の王), 33=BT03(角王大出陣!!)。`card_kind`: 1=モンスター,2=魔法,3=アイテム,4=必殺技,5=フラッグ。

### 1-2. スケルトン化（メタ＋公式rulesを正本に）
- 取得データ → `{id, no, name, type, world, attributes, rarity, size/power/critical/defense, rules}` の骨組みJSONへ。
  - `type`: モンスター→`monster` / 必殺技→`impact` / 魔法→`spell` / アイテム→`item`。
  - `id`: `bf-h-eb01-0002` のように `製品プレフィックス-4桁`。
  - **`rules`（表示用効果文＝公式テキスト）を必ず入れる**。空だとUIに「能力なし」表示になる（[[card-rules-display-field]]）。

### 1-3. DSLエンコード（並列Workflow・バッチ化）
- **カードは公式テキスト通りに実装**（[[card-impl-official-text]]）。要約や独自解釈で効果を変えない。例外はユーザー明示時のみ。
- **汎用プリミティブで表現し、カード名ハードコード禁止**（[[card-impl-generic-primitives]]）。「◯◯の効果以外で発動不可」等も条件付き発動可能として条件/フラグで表す。
- **1枚1エージェントにしない。~8-12枚/エージェントでバッチ化**（[[workflow-batch-cards]]）。EB01では ultracode Workflow で **5分割**した（※ユーザー指定でカード分割は最大5だった。指定が無ければ8-12枚/agentを基準に）。
- Workflow の作り: `pipeline`/`parallel` で各バッチ agent が「スケルトン＋公式テキスト＋既存op語彙チートシート＋実例カードファイル＋src/13-15」を読み、`{cards, newOps, notes}` を StructuredOutput で返す。埋め込みデータは `args` でなくスクリプトに直書きが安全（`args`が渡らない不具合を踏んだ）。合成時はスケルトンのメタを正本にし、agentのDSLキーだけ重ねる（メタ改変を防ぐ）。
- **既存opで表せるなら新opを作らない**。op語彙は `grep -rhoE '=== "[a-zA-Z]+"' src/13-abilities-core.js src/14-ability-script.js src/15-ability-effects.js src/04-cost-resource.js src/05-stats.js` で抽出できる（EB01実装時はチートシート化して各agentに渡した）。

### 1-4. 新規opの実装（自分＝メインループで。並列agentにやらせない）
- 新opは責務のモジュールへ（下表）。汎用プリミティブとして filter/zones/amount 等をJSONで指定できる形にする。
- EB01で追加した約20opが直近の実例。`git show 746feb1`（EB01）を読むと、条件/効果/スクリプト/コスト/継続/イベント各種の追加パターンが分かる。
- **必ず実操作経路の回帰テストを追加**（CLAUDE.md必須）: `useCardAction`/`callMonster`/`resolvePendingResolution`/`executeAbilityBody` 等を通す。効果本体だけのテストは不可。`tests/effects-regression.test.js` に `load<製品>CardById` ローダとテストを足し、末尾IIFEに登録。

### 1-5. 登録・検証・コミット
- `data/cardsets.json` に `{id, name, generation:"100"(ハンドレッド)/"無印"等, file}` を追加。カードファイルは cardsets 経由で読み込まれる（追加ローダ不要。src/モジュールを増やした時だけ `index.html`/`play.html`/`tests/*-browser-smoke.html` のローダに<script>追加。※`netplay.html` は旧・中継版で現在は非使用＝保守不要）。
- 検証（§4）→ コミット → main へ ff（§5）→ 画像取得（§3）→ zip（§5）。

### 1-6. 構築済みデッキ（デッキレシピ）データの取得・追加
※「デッキ作る時のデータ」＝公式/有志の**構築済みデッキレシピ**をアプリに載せる手順。カードデータ(§1-1〜1-5)とは別物。

1. **デッキコードの入手**: 構築済みデッキ情報の出典は **note.com/sigure_puyopuyo/n/ne41f75894c59**（各製品ぶんのDECK LOGコードが載っている。例: H-SD01紅蓮の拳=`11VTY`）。[[deck-data-source]]。
2. **レシピ取得**: `python3 tools/fetch_card_data.py --deck <CODE>` — DECK LOG API `decklog.bushiroad.com/system/app/api/view/<CODE>` からフラッグ＋カード枚数リストを取得（`{flag, list:[{card_number,num,name,card_kind,size}]}`）。`--expansion <N> --deck <CODE>` にすると製品カード取得と枚数統合ができる。詳細はツール冒頭docstringと [[bf-card-data-fetch-method]]。
3. **デッキJSONを作る**: `data/decks/<id>.json` は `{schemaVersion, decks:[{id, name, flag, buddy, recipe:[[cardId, count], ...]}]}`（1ファイルに複数デッキ可）。`flag`/`buddy`/`recipe` の各cardIdはこのアプリの `card.id`（フラッグは `dragon-world` 等の flag id）。DECK LOG の card_number → `card.id` へ対応付ける。既存 `data/decks/*.json` を手本に。
4. **登録**: `data/decksets.json` の `sets[]` に `{id, name, file:"data/decks/<id>.json"}` を追加。`loadGameData`（`src/02`）が decksets 経由で `deckProfiles` に load し、builder のプリセット/対戦のデッキ選択に出る。
5. 検証: builder(`builder.html`)や対戦でデッキが選べること、`deckReferencesKnown`（未実装カード参照が無いこと）を確認。デッキは §1 のカード実装が済んでいる製品ぶんだけ追加する（未実装カードを含むデッキは弾かれる）。

## 2. エンジン設計の勘所（新opを足す時に知るべきこと）

### 2-1. カードDSLと配置
- カード: `{cost/costs, callCost/castCost/equipCost, conditions/callConditions/equipConditions/useConditions, target, effects, script, keywords, continuous, soulContinuous, abilities[{kind:"triggered"|"activated"|"impact"|"spell", event/timing, conditions, cost, effects|script}], 各種フラグ(cannotCallZones/destroyImmunity/cannotReturnToHand/uniqueSet/nullifyImmune/equipOnlyByAbility/cannotCallNormally/deckAnyFlag 等)}`。
- op を書く場所:
  - 条件 `checkCondition` — `src/13-abilities-core.js`。
  - 効果op `executeAbilityEffect` — `src/15-ability-effects.js`。
  - スクリプトop `executeAbilityScriptStep` — `src/14-ability-script.js`（selectCards/moveSelected/ifCondition/chooseBranch 等の対話系）。
  - コストop — `src/04-cost-resource.js`（canPay/pay/payWithSelection の3経路すべてに足す）。
  - 継続・ステータス（modifyStats/soulCount/soulStatSum/keyword付与）— `src/05-stats.js`, キーワードは `src/18-tooltip-format.js` の `hasKeyword`。
  - 破壊/ターン処理・破壊置換 — `src/11-destroy-turn.js`。攻撃/連携/イベント発火 — `src/09-attack.js`。
- フィールドイベントは `runFieldEventTriggers(base, owner)` が `allyBase`/`opponentBase` に接頭辞付与（例 `drawByEffect`→`allyDrawByEffect`）。`enter`/`attack`/`attacked`/`destroyByAttack` はカード自身へ直接発火（接頭辞なし）。

### 2-2. 対抗（たいこう）ウィンドウ ＝ `state.pendingAction` / `state.pendingAttack`（重要）
- 任意で使うカード/能力の**宣言時に相手へ対抗機会を与える**のが正しい挙動（2018年6月以前ルール）。`beginPendingAction({kind, owner, responder, card, ability, ...})`（`src/07-actions-turn.js`）で保留、`resolvePendingResolution`→`resolvePendingAction`が`kind`で分岐（call/spell/impact/setSpell/equip/ability）。
- **落とし穴**: `useHandAbilityAction`（`src/13`）は元々 spell/impact しか対抗ウィンドウを開かず、**手札発動の起動能力（変身/搭乗の hand版, kind:"activated"）が即実行**され対抗が出なかった。`793e4ca` で activated も `kind:"ability", fromHand:true` で保留するよう修正し、`resolvePendingAbility`（`src/07`）を手札発動対応（ドロップ着地・equipSelf回収・callSelfFromHand ロールバック）に拡張。**新しく「手札から使う能力」を足す時はこの経路に載っているか確認**。

### 2-3. カード画像＝製品ごと base64 パック（[[card-images-feature]]）
- 個別webpは廃止。`data/images/{pack}.imgpack.json`（pack名=カードJSONファイル名stem=cardset id）に集約。描画は `createCardElement`（`src/12`）に集約され index/netplay/権威版(play.html)が共有。必要な製品パックだけ遅延fetch（`ensureImagePackLoaded`/`cardIdToPack`、builderは`builderFillThumbs`）。フォールバック: パックdataURL→公式URL→名前プレースホルダ。
- **必殺技/必殺モンスターは公式カードが横長** → 描画に `landscape` クラス＋横長比フレーム（画像は既に横長保存、回転不要）。

### 2-4. クライアント3種
- `index.html`（ローカル・フルエンジン）、`play.html`（権威版シンクライアント・`__BUDDYFIGHT_THIN__`、src/全モジュール＋`play.js`）。両者とも `createCardElement` 等エンジン描画を共有するので、描画系の修正は両クライアントに一括で効くことが多い。※`netplay.html`（旧・中継版）も同エンジンを共有するが現在は非使用。
- **UI操作は3クライアントで統一済み**（`79cbb25`、2026-07-02）: カード操作＝「カードタップ→下部共有メニュー `showActionMenu`（src/20）」、本体攻撃＝相手装備枠タップ、候補外タップ無視、確認ダイアログなし、カードシートは閲覧専用。操作パネルのカード系ボタン/対象`<select>`はCSSで全クライアント非表示（selectは隠し値置き場として存続＝エンジンが `elements.effectTarget.value` 等を読むため**DOMからは消さないこと**）。攻撃ハイライトはクラス2系統（modular=`attack-target-candidate`/play.js=`attack-target-highlight`）のまま見た目を赤パルスに統一。バディコール宣言は `selectHandCard` が「同一カード再選択では維持」する（メニュー方式の宣言→再タップ→コールを成立させるため）。選択プロンプトは権威版も `#selectionDialog` を共用（staleなdialog closeイベントは open ガードで無視。src/16）。

## 3. カード画像の手順

1. `python3 tools/fetch_card_images.py` — 公式画像URLをカード番号から導出（`H-EB01/0002`→`heb_01_0002.png`）、ffmpeg(libwebp)で幅160pxのWebPに縮小して `data/images/{id}.webp` に保存。Sレア/再録で導出名が違う場合は詳細ページから実URLをスクレイプ、非数値番号(S002等)も対応。要 ffmpeg（Pillow不要）。
2. `python3 tools/build_image_packs.py --prune` — 個別webpを製品ごと base64 パックに集約し、個別webpを削除。
- 詳細はツール冒頭docstringと [[card-images-feature]]。

## 5. Git / zip 運用

- **git worktree で作業**（このセッションは `.claude/worktrees/prompt-roundtrip-slice1`）。Edit/Write は worktree パスに。Bashで`cd`する時はMAINでなくworktreeへ（MAINで`git add -A`すると未追跡ジャンクを巻き込む事故あり。実際に一度踏んだ）。
- コミット→ `git -C <MAIN> merge --ff-only worktree-prompt-roundtrip-slice1` で main へ ff。GitHubリモート/認証は無い。
- **zip運用**（[[zip-packaging-policy]]）: 原則**差分zip（増分＝前回zip以降）**。フルzipはユーザー明示指示時のみ。階層は元と同じ `buddyfight-app/<相対パス>`。基準コミットはzipを渡すたびに前進（メモリ末尾で追跡）。削除は差分zipに表せないので都度メッセージで明示。現状の最終フル=`buddyfight-app_2026-07-01.zip`(=`b982825`, 画像パック構成104エントリ)。
- 上書き/削除前は必ず前版をバックアップ（[[keep-previous-versions]]、`.bak`運用）。

## 6. 落とし穴・決定事項（ハマりどころ）

- **カード実装ドキュメントは一部古い**（[[card-impl-docs-outdated]]）。HANDOFF/旧ログは鵜呑みにせず要確認。
- センシティブbashフック: python内の `os.environ['CLAUDE_JOB_DIR']` 直書きはブロックされる。bashの`$CLAUDE_JOB_DIR`展開かargvで渡す。
